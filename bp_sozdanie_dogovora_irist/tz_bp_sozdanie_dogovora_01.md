# ТЗ: доработка бизнес-процесса создания договора и дополнительного соглашения в Битрикс24

Источник: скан `Тех. зад. доработка Битрикс 2026_01.pdf`.

Примечание по транскрибации: документ является рукописным сканом. Текст восстановлен вручную по изображению. Неоднозначные места отмечены как требующие уточнения.

---

## 1. Транскрибация исходного документа

### 1.1. Создание договора

Создание договора.

- Срок начала работ / этапа.
- Срок окончания работ / этапа.
- Предусмотреть добавление сроков начала / окончания этапов.
- Уведомление по срокам.

Статус оформления:

- Отправлен по ЭДО.
- Подписан по ЭДО.

Оригинал / копия:

- Скан-копия.

Оплата по договору:

- Определенный срок.
- Сумма.
- Дата.
- Уведомление заранее.

Там, где вставляют ИНН, добавить графу для ввода ОГРНИП для ИП.

В CRM отсутствует сквозная автоматическая нумерация договоров.

### 1.2. Создание дополнительного соглашения

Создание / добавление дополнительного соглашения.

1. В название ДС / в шапку карточки не вставляется № ДС.
   - Примечание из скана: "ставится при создании только внешний номер" / формулировка требует уточнения.

2. Добавить по аналогии с договорами:
   - Цена.
   - Оплата.
   - Предоплата.
   - Даты начала / окончания.
   - Статус оформления.
   - Оригинал / копия.
   - Место хранения.

Необходимо улучшить поиск:

1. Из раздела "Контрагенты" / "Компании".
2. Из "Договора".

А именно: при поиске по контрагенту не находит его в базе "Компании" и / или не находит заключенный с ним договор. Проблема возникает довольно часто.

## 2. Цель доработки

Настроить в Битрикс24 удобный и управляемый процесс создания, учета и контроля договоров и дополнительных соглашений, чтобы пользователи могли:

- фиксировать сроки договора и отдельных этапов;
- контролировать оплату по датам и суммам;
- видеть статус оформления и подписания;
- хранить скан-копии и данные об оригиналах;
- быстро находить контрагентов, договоры и дополнительные соглашения;
- использовать сквозную автоматическую нумерацию договоров;
- корректно работать с ИП через ИНН и ОГРНИП.

Применимость: высокая для компаний, где договоры и ДС ведутся в CRM и требуют контроля сроков, оплат и документооборота.

Улучшения / альтернативы: если договоров много, рекомендуется использовать отдельную смарт-процесс-сущность "Договоры", а не только поля в сделке. Это упростит поиск, права доступа, отчеты и связь "Компания - Договор - Дополнительное соглашение".

---

## 3. Объекты CRM

### 3.1. Договор

Рекомендуемая реализация: отдельная сущность CRM "Договоры" через смарт-процесс Битрикс24.

Карточка договора должна быть связана с:

- компанией / контрагентом;
- контактом, если договор заключается с физическим лицом или ИП;
- сделкой, если договор создается в рамках продажи;
- дополнительными соглашениями;
- файлами договора и скан-копиями.

Применимость: высокая, если договор может жить дольше одной сделки или иметь несколько ДС, оплат и этапов.

Улучшения / альтернативы: для простого процесса можно хранить договор как элемент сделки, но это хуже масштабируется при повторных договорах с одним контрагентом.

### 3.2. Дополнительное соглашение

Рекомендуемая реализация: отдельная сущность CRM "Дополнительные соглашения" или дочерний смарт-процесс, связанный с договором.

Карточка ДС должна быть связана с:

- основным договором;
- компанией / контрагентом;
- сделкой, если ДС относится к конкретной сделке;
- файлами ДС и скан-копиями.

Применимость: высокая для случаев, когда к одному договору создается несколько дополнительных соглашений.

Улучшения / альтернативы: если ДС создаются редко, допустимо хранить их как связанные элементы / документы внутри карточки договора, но тогда сложнее вести поиск, статусы и аналитику.

## 4. Требования к карточке договора

### 4.1. Сроки договора и этапов

Добавить в карточку договора поля:

- дата начала работ;
- дата окончания работ;
- дата начала этапа;
- дата окончания этапа;
- описание / наименование этапа;
- ответственный за этап;
- статус этапа.

Если этапов может быть несколько, реализовать их как множественный список, связанную таблицу или отдельный дочерний смарт-процесс "Этапы договора".

Настроить уведомления:

- до наступления даты начала этапа;
- до наступления даты окончания этапа;
- при просрочке этапа;
- ответственному менеджеру;
- руководителю, если просрочка превышает согласованный лимит.

Применимость: высокая для договоров с поэтапными работами, поставками, услугами или проектным исполнением.

Улучшения / альтернативы: если этапы имеют сложную логику исполнения, лучше использовать задачи или проектные стадии, связанные с договором. Если нужны только контрольные даты, достаточно полей и роботов CRM.

### 4.2. Статус оформления договора

Добавить поле "Статус оформления" со значениями:

- черновик;
- на согласовании;
- отправлен по ЭДО;
- подписан по ЭДО;
- подписан на бумаге;
- отклонен / требует правок;
- архив.

Обязательные значения из скана:

- отправлен по ЭДО;
- подписан по ЭДО.

Применимость: высокая для контроля документооборота и понимания, можно ли начинать исполнение договора.

Улучшения / альтернативы: при интеграции с оператором ЭДО статусы лучше обновлять автоматически через API / вебхуки, а не вручную менеджером.

### 4.3. Оригинал, копия и скан-копия

Добавить поля:

- наличие оригинала: да / нет;
- тип хранения: оригинал / копия / скан-копия;
- файл скан-копии;
- место хранения оригинала;
- дата получения оригинала;
- ответственный за хранение.

Применимость: высокая для организаций, где требуется контроль бумажных оригиналов.

Улучшения / альтернативы: для электронного документооборота можно сделать "скан-копию" необязательной, если документ подписан через ЭДО и файл подписи хранится в карточке.

### 4.4. Оплата по договору

Добавить блок "Оплата по договору":

- сумма платежа;
- дата платежа;
- тип платежа: предоплата / постоплата / этапный платеж / окончательный расчет;
- статус оплаты: ожидается / оплачено / просрочено / отменено;
- дата фактической оплаты;
- комментарий;
- уведомление заранее.

Если платежей несколько, реализовать оплату как множественный блок или отдельный дочерний смарт-процесс "Платежи по договору".

Настроить уведомления:

- за N дней до плановой даты оплаты;
- в день плановой оплаты;
- при просрочке оплаты;
- ответственному менеджеру и / или бухгалтерии.

Применимость: высокая для договоров с графиком платежей, предоплатой и несколькими суммами.

Улучшения / альтернативы: если в компании уже используется 1С или другая учетная система, статус фактической оплаты лучше получать интеграцией, а в Битрикс24 хранить план и контроль.

### 4.5. ИНН и ОГРНИП

В местах, где пользователь вводит ИНН контрагента, добавить возможность ввода ОГРНИП для индивидуальных предпринимателей.

Рекомендуемые правила:

- ИНН ИП: 12 цифр;
- ОГРНИП: 15 цифр;
- поле ОГРНИП показывать для типа контрагента "ИП";
- для юридических лиц использовать ОГРН / КПП по стандартной логике реквизитов.

Применимость: высокая, если компания работает с ИП.

Улучшения / альтернативы: по возможности использовать стандартные реквизиты Битрикс24, а не дублирующие пользовательские поля. Это снизит риск расхождений в документах и интеграциях.

### 4.6. Сквозная автоматическая нумерация договоров

Настроить автоматическую сквозную нумерацию договоров.

Требования:

- номер присваивается автоматически при создании или переводе договора в согласованную стадию;
- номер уникален в рамках всех договоров;
- формат номера согласовать с заказчиком;
- номер не должен сбрасываться внутри разных менеджеров, сделок или компаний;
- при удалении / отмене договора номер не переиспользуется без отдельного решения администратора.

Рекомендуемый формат:

```text
ДГ-YYYY-000001
```

Где:

- `ДГ` - префикс договора;
- `YYYY` - год создания;
- `000001` - сквозной порядковый номер.

Применимость: высокая, если договоры должны иметь юридически понятную и единую нумерацию.

Улучшения / альтернативы: можно сделать отдельные префиксы по типам договоров, направлениям бизнеса или юрлицам, но это усложнит правила и требует согласования с бухгалтерией / юристами.

## 5. Требования к карточке дополнительного соглашения

### 5.1. Наименование и номер ДС

Исправить формирование названия карточки ДС.

Требование:

- в название / шапку карточки ДС должен автоматически подставляться номер ДС;
- ДС должно быть связано с основным договором;
- в названии желательно отображать номер ДС и номер основного договора.

Рекомендуемый формат:

```text
ДС №{номер ДС} к договору №{номер договора} от {дата договора}
```

Требует уточнения:

- какой номер считать номером ДС: внутренний номер Битрикс24, внешний номер, ручной номер из документа или автоматически созданный номер;
- нужно ли использовать отдельную сквозную нумерацию для ДС.

Применимость: высокая, так как корректное название карточки напрямую влияет на поиск и работу пользователей.

Улучшения / альтернативы: если ДС печатается из шаблона документа, один и тот же номер должен использоваться и в карточке CRM, и в печатной форме.

### 5.2. Поля ДС по аналогии с договором

Добавить в карточку ДС поля:

- цена / сумма изменения;
- условия оплаты;
- предоплата;
- дата начала действия;
- дата окончания действия;
- статус оформления;
- оригинал / копия / скан-копия;
- место хранения;
- файл ДС;
- комментарий.

Применимость: высокая, если ДС меняет сумму, сроки, порядок оплаты или условия исполнения договора.

Улучшения / альтернативы: часть полей можно наследовать из основного договора, чтобы пользователю не приходилось заполнять одно и то же повторно. При этом изменения по ДС нужно хранить отдельно.

## 6. Улучшение поиска

### 6.1. Поиск контрагентов / компаний

Проблема из скана: при поиске по контрагенту система не всегда находит компанию в базе "Компании".

Требования:

- поиск должен работать по названию компании;
- поиск должен работать по ИНН;
- поиск должен работать по ОГРНИП для ИП;
- поиск должен учитывать альтернативные написания и частичное совпадение;
- при создании договора пользователь должен быстро выбрать существующую компанию / контрагента.

Применимость: высокая, так как плохой поиск приводит к дублям компаний и ошибкам в договорах.

Улучшения / альтернативы: дополнительно настроить контроль дублей по ИНН / ОГРНИП и регламент слияния дублей.

### 6.2. Поиск договоров

Проблема из скана: при поиске по контрагенту система не всегда находит заключенный с ним договор.

Требования:

- из карточки компании должен быть виден список связанных договоров;
- в карточке договора должна быть обязательная связь с компанией / контрагентом;
- поиск договоров должен работать по номеру договора, названию, компании, ИНН, ОГРНИП и статусу;
- при создании ДС пользователь должен выбирать основной договор из списка договоров выбранного контрагента.

Применимость: высокая для компаний с повторными договорами и несколькими ДС.

Улучшения / альтернативы: создать отдельный список / смарт-процесс "Договоры" с фильтрами по компании, статусу, срокам и оплатам. Это надежнее, чем искать договоры только через текстовые поля сделки.

## 7. Роботы, триггеры и автоматизация

Настроить автоматизацию:

- присвоение номера договора;
- формирование названия договора;
- формирование названия ДС;
- уведомления по срокам этапов;
- уведомления по датам оплаты;
- уведомления по статусам ЭДО;
- контроль обязательных полей перед переходом на финальные стадии;
- постановка задач ответственным при просрочке сроков / оплат.

Применимость: высокая при регулярном потоке договоров.

Улучшения / альтернативы: если стандартных роботов Битрикс24 недостаточно для нумерации и сложных связей, использовать REST API, вебхуки или бизнес-процессы.

## 8. Обязательные поля и проверки

Перед сохранением / переводом договора в работу проверять:

- выбран контрагент;
- заполнен номер договора;
- заполнены даты начала и окончания;
- заполнен статус оформления;
- приложен файл договора или указана причина отсутствия;
- для ИП заполнен ОГРНИП, если он требуется процессом;
- заполнен график оплат, если договор предусматривает оплату.

Перед сохранением / переводом ДС в работу проверять:

- выбран основной договор;
- заполнен номер ДС;
- заполнена дата ДС;
- заполнен статус оформления;
- приложен файл ДС или указана причина отсутствия.

Применимость: высокая для снижения ошибок пользователей.

Улучшения / альтернативы: обязательность полей можно делать не сразу при создании, а по стадиям. Это удобнее, если карточка сначала создается как черновик.

---

## 9. Критерии приемки

Доработка считается выполненной, если:

1. В карточке договора доступны поля сроков договора и этапов.
2. По срокам договора / этапов приходят уведомления ответственным.
3. В статусах договора есть значения "Отправлен по ЭДО" и "Подписан по ЭДО".
4. В карточке договора можно хранить скан-копию и данные об оригинале / копии.
5. В карточке договора можно указать график оплат с суммой, датой и уведомлением заранее.
6. Для ИП доступно поле ОГРНИП в зоне ввода реквизитов.
7. Договоры получают уникальную сквозную автоматическую нумерацию.
8. В карточке ДС автоматически отображается номер ДС в названии / шапке.
9. В карточке ДС доступны поля цены, оплаты, предоплаты, дат, статуса, оригинала / копии и места хранения.
10. Поиск находит компанию / контрагента по названию, ИНН и ОГРНИП.
11. Поиск находит договоры, связанные с выбранным контрагентом.
12. Из карточки компании видны связанные договоры и ДС.

6. Сколько этапов может быть у договора: один, несколько, неограниченно?
